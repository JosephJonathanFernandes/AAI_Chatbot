\chapter*{Abstract}
\addcontentsline{toc}{chapter}{Abstract}
This report presents a college-domain conversational assistant that combines local multi-signal NLU control (preprocessing, intent, scope, emotion, and context) with external LLM response generation. Component-level evaluation on repository intent patterns shows strong semantic-routing performance (accuracy 0.783, macro F1 0.756), while end-to-end evaluation on 120 scenarios captures deployment-facing behavior under noisy, vague, emotional, and out-of-scope (OOS) inputs. A key finding is that a substantial portion of observed failures in batch execution are infrastructure-related (API rate-limit/provider failures) rather than core routing logic errors. Separating infrastructure failures from functional failures yields a clearer estimate of system behavior and supports more accurate interpretation of confidence intervals, OOS gating quality, and latency trade-offs.
The primary contribution is a deployment-realistic evaluation methodology that separates infrastructure failures from functional routing errors, yielding corrected confidence intervals that better reflect core assistant logic.

\chapter{Literature Survey}
This chapter focuses on five high-impact studies that align directly with the assistant design goals: contextual understanding, efficient intent routing, out-of-scope rejection, and safe text generation. Each selected paper contributes a different system requirement, so together they form a compact but complete foundation for this project.
Rather than listing many partially related works, this survey prioritizes depth on a small set of papers that map directly to implemented design choices in this report. The sequence is intentionally structured from representation learning foundations to deployment constraints and finally to generation safety.

\subsection{The Paradigm Shift to Self-Attention Architecture: Vaswani et al., 2017}
RNN-based language systems process tokens step-by-step, which often weakens long-range dependency capture and limits parallelization. Vaswani et al. introduced the transformer, where multi-head self-attention and positional encoding replace recurrent computation \cite{vaswani2017attention}. This redesign improved scalability and contextual modeling quality, while introducing the known quadratic attention cost $O(n^2)$. In this project, the paper provides the architectural rationale for using transformer-era semantic representations instead of purely sequential NLU encoders.
From a systems perspective, this contribution is important because intent routing quality in noisy conversational settings depends on context-rich token interactions, not only local word order. The transformer formulation therefore supports the report's decision to rely on embedding-driven understanding before downstream control decisions.

\subsection{Bidirectional Pre-Training for Semantic Context: Devlin et al., 2019}
Devlin et al. showed that bidirectional pre-training with masked language modeling can produce robust representations that transfer effectively to downstream tasks \cite{devlin2019bert}. BERT demonstrates that large-scale unsupervised pre-training substantially improves semantic feature quality before task adaptation. For the current assistant, this supports encoding noisy user input into a richer semantic space prior to intent decisions. The paper also implies an important limitation for deployment: representation strength alone does not enforce domain boundaries or reliable out-of-scope behavior.
This observation is especially relevant for college-assistant use cases, where semantically similar but policy-different requests can occur in the same conversation. As a result, this work is used here as representation justification, while control logic is handled separately through thresholds and scope checks.

\subsection{Lightweight Intent Routing under Deployment Constraints: Casanueva et al., 2020}
Casanueva et al. emphasize practical intent detection with sentence-embedding retrieval rather than heavyweight classifier stacks \cite{casanueva2020banking77}. By comparing user embeddings with intent prototypes through cosine similarity, their method achieves strong routing speed with competitive accuracy. This idea maps directly to the project implementation, where similarity-based semantic routing is combined with lexical evidence to maintain responsiveness under real-time constraints.
For deployment-oriented assistants, this contribution is significant because it demonstrates that efficient routing can be achieved without sacrificing robustness to paraphrases. It also supports an architecture where fast semantic matching is used as a first-pass decision mechanism that can be refined by additional control signals.

\subsection{Benchmarking and Mitigating Out-of-Scope Queries: Larson et al., 2019}
Larson et al. made out-of-scope handling measurable by introducing CLINC150 with explicit OOS samples \cite{larson2019clinc150}. Their benchmark highlights a key reliability issue: closed-set models may confidently assign a wrong in-domain label when no valid label exists. This project adopts that lesson through threshold-based acceptance rules and semantic scope checks that prefer clarification or fallback over forced intent assignment.
This directly addresses a common user-trust failure mode in student support assistants: confident but irrelevant answers. By prioritizing safe rejection and clarification paths, the design follows a deployment-centered interpretation of model quality rather than a closed-world accuracy-only interpretation.

\subsection{Aligning Generative Output for Conversational Safety: Ouyang et al., 2022}
Ouyang et al. describe RLHF as a practical alignment pipeline that improves instruction following and reduces harmful output tendencies in large language models \cite{ouyang2022rlhf}. The approach combines supervised fine-tuning, a preference-trained reward model, and policy optimization. For this assistant, the key implication is that alignment improves behavior quality but does not by itself guarantee domain-accurate facts. Therefore, generation is constrained by grounding and policy controls tied to college-specific data.
In practical terms, this means generation quality must be interpreted together with factual anchoring mechanisms. The report therefore treats aligned generation as a necessary layer for tone and instruction compliance, but not as a replacement for explicit domain constraints.

Together, these five papers provide a complete and non-overlapping theoretical basis for the implemented assistant: transformer foundations, contextual encoding, efficient intent inference, OOS reliability, and aligned generation safety.

\chapter{Methodology and Design}
The system is designed as a domain-constrained conversational assistant that combines a structured NLU/control layer with a controlled response generation layer. This hybrid approach follows system-level findings that robust assistants require orchestration, context handling, uncertainty management, and evaluation discipline in addition to fluent text generation \cite{caldarini2022literature,gao2020recent,gao2019neural,khatri2018alexa,maroengsit2019evaluation}.

\subsection{System Overview}
The design targets a college-information domain and is organized as a multi-signal pipeline:
\begin{itemize}
	\item \textbf{Control and understanding:} preprocessing, intent inference, uncertainty estimation, scope (in-scope/OOS) decisions, and conversation context.
	\item \textbf{Response layer:} knowledge-grounded response planning followed by controlled natural language generation.
	\item \textbf{Monitoring and evaluation:} telemetry signals (intent confidence, latency, error modes) and test-driven regression checks to track reliability.
\end{itemize}
This decomposition reflects common taxonomies for conversational agents and supports predictable behavior under ambiguous or out-of-scope inputs \cite{hussain2019survey,larson2019clinc150,casanueva2020banking77}.

\subsection{Data Flow Pipeline}
The end-to-end pipeline is defined as a deterministic sequence of transformations and decisions:
\begin{center}
\textbf{Input $\rightarrow$ Preprocessing $\rightarrow$ Representation $\rightarrow$ Intent/Scope Decision $\rightarrow$ Response Planning $\rightarrow$ Generation $\rightarrow$ Output}
\end{center}

More explicitly:
\begin{enumerate}
	\item \textbf{Input:} user utterance $u_t$.
	\item \textbf{Preprocessing:} normalization to reduce sensitivity to character-level noise that can degrade NLU performance \cite{belinkov2018synthetic,sun2020adversarial}.
	\item \textbf{Representation:} a lexical vector $v(u_t)$ (sparse) and a semantic embedding $e(u_t)$ (transformer encoder) \cite{devlin2019bert,liu2019roberta}.
	\item \textbf{Intent inference:} compute intent scores and aggregate them into a calibrated posterior $p(y\mid u_t)$ over intents.
	\item \textbf{Uncertainty and scope:} compute confidence $c=\max_y p(y\mid u_t)$ and apply thresholds to accept, clarify, or reject (OOS) \cite{larson2019clinc150,casanueva2020banking77}.
	\item \textbf{Response planning:} condition on $(u_t,m_t)$ and retrieved facts $k_t$ to form a response plan that constrains generation.
	\item \textbf{Generation and output:} produce the final natural language response with conservative behavior under uncertainty \cite{radford2019gpt2,ouyang2022rlhf,thoppilan2022lamda}.
\end{enumerate}

\subsection{Model Design: Intent Classification and Response Generation}
\subsubsection{Intent Classification}
Intent classification is modeled as multi-class prediction over a closed intent set. Given tokenized input $x=[x_1,\ldots,x_n]$, a transformer encoder produces contextual states $H=[h_1,\ldots,h_n]$ \cite{vaswani2017attention,devlin2019bert}. A pooled vector $z$ is mapped to a posterior over intents:
\[
p(y\mid x)=\mathrm{softmax}(Wz+b),\quad c=\max_y p(y\mid x).
\]
For efficiency, inference may be complemented with similarity-based retrieval in embedding space, where each intent $i$ has a prototype text $t_i$ and score
\[
s_i=\cos\bigl(e(u_t),e(t_i)\bigr),\quad \hat{y}=\arg\max_i s_i.
\]
The final decision uses uncertainty-aware thresholds (e.g., reject if $c<\tau$ or $\max_i s_i<\gamma$), reflecting the need for calibrated OOS behavior \cite{larson2019clinc150,casanueva2020banking77,zhang2020fewshot}. In the implemented system, $\tau$ is intent-specific via dynamic confidence thresholds (typical academic intents in the 0.45--0.60 range from \texttt{confidence\_threshold\_manager.py}), while the semantic scope gate uses $\gamma=0.50$ (from \texttt{scope\_detector.py}); this setting directly affects observed OOS precision/recall.

\subsubsection{Response Generation}
Response generation is treated as conditional text generation using an autoregressive decoder-style language model (LM). Given a structured response plan and grounding context, generation models the probability of an output sequence $r=[r_1,\ldots,r_T]$ as
\[
p(r\mid u_t,m_t,k_t)=\prod_{t=1}^T p(r_t\mid r_{<t},u_t,m_t,k_t).
\]
Decoder-only pre-training establishes the base generative capability \cite{radford2018gpt,radford2019gpt2,brown2020fewshot}. Alignment and dialogue specialization approaches improve instruction adherence and conversational quality \cite{ouyang2022rlhf,thoppilan2022lamda,openai2022chatgpt}, but they do not guarantee domain factuality; therefore, grounding and scope control remain first-class design requirements \cite{roller2021recipes,khatri2018alexa}.

\subsection{Algorithms and Models Used: RNN vs Transformer Comparison}
Traditional joint NLU systems often rely on recurrent encoders (BiRNN/BiLSTM) trained jointly for intent and slots \cite{liu2016joint,hakkani2016multidomain}. Recurrence provides a strong inductive bias for local sequential patterns but is inherently sequential and can limit throughput.

Transformer encoders replace recurrence with self-attention, enabling parallel training and stronger long-range context modeling \cite{vaswani2017attention}. Pre-trained transformer encoders (e.g., BERT/RoBERTa) provide transferable representations that improve intent classification and joint intent/slot baselines \cite{devlin2019bert,liu2019roberta,chen2019bertjoint}.

For response generation, decoder-style transformers support fluent multi-turn responses but require alignment and system constraints to mitigate hallucination risk \cite{radford2019gpt2,ouyang2022rlhf}. End-to-end trainable dialogue systems have also been explored, including task-oriented dialogue networks and memory-based approaches \cite{wen2017network,bordes2017endtoend}; however, these formulations provide fewer explicit control points for domain scoping and safety policies.

\subsection{Design Justification}
The design adopts a modular, hybrid architecture rather than a fully end-to-end conversational model because modular decomposition improves controllability and allows explicit enforcement of domain scope, uncertainty handling, and evaluation checkpoints \cite{hussain2019survey,gao2019neural,khatri2018alexa,roller2021recipes}.

Uncertainty-aware intent routing is emphasized because deployed assistants must handle ambiguous and out-of-scope inputs; explicit OOS gating is supported by intent benchmarks designed to measure rejection performance \cite{larson2019clinc150,casanueva2020banking77}. A hybrid intent stack (lexical evidence + transformer-derived semantics) supports a practical trade-off among interpretability, latency, and paraphrase robustness \cite{devlin2019bert,liu2019roberta}.

Robustness and measurement risks motivate conservative preprocessing and careful interpretation of automated evaluation results. Neural NLU brittleness under noise and dataset artifacts can inflate reported performance without improving real-world reliability \cite{belinkov2018synthetic,sun2020adversarial,gururangan2018annotation,geva2019shortcut,swayamdipta2020dataset}. The methodology therefore prioritizes scope control and grounded generation over unconstrained fluency, consistent with survey evidence that reliable chatbot performance depends on system integration and evaluation discipline \cite{caldarini2022literature,maroengsit2019evaluation}. The following section maps this design to concrete modules and execution pathways in the repository.

\chapter{Implementation}
This section operationalizes the design using Python modules in the repository. The implementation emphasizes reproducible preprocessing, explicit scope and confidence controls, and a monitored LLM response layer.

\subsection{Development Environment: Languages, Tools, and Libraries}
\textbf{Language and runtime.} The system is implemented in Python and executed as either a Streamlit web application (interactive UI) or a command-line runner.

\textbf{Core libraries.}
\begin{itemize}
	\item \textbf{UI:} Streamlit for the chat interface and session state management.
	\item \textbf{NLU models:} PyTorch as the deep learning backend, Hugging Face Transformers for transformer pipelines, and Sentence-Transformers for embedding-based semantic similarity.
	\item \textbf{Classical ML:} scikit-learn for TF--IDF vectorization and logistic regression.
	\item \textbf{Data and utilities:} NumPy/Pandas for numeric and tabular handling; Requests for HTTP.
	\item \textbf{LLM providers:} Groq client/HTTP for the primary model endpoint and \texttt{google-genai} for Gemini fallback; \texttt{python-dotenv} for environment-variable configuration.
	\item \textbf{Persistence:} SQLite (via the Python standard library) for interaction logging and lightweight analytics.
\end{itemize}

\textbf{Testing and quality tools.} The development toolchain includes PyTest (with coverage, parallelism, and timeouts), Hypothesis for property-based tests, Locust for load/performance testing, and formatting/linting tools (Black, Flake8, isort).

\subsection{Dataset Description: Source and Preprocessing Steps}
Two local JSON artifacts serve as the primary data sources:
\begin{itemize}
	\item \textbf{Intent dataset} (\texttt{data/intents.json}): an intent schema containing tags, paraphrased training patterns, and optional keyword lists. This file is used to train lightweight intent components at application startup.
	\item \textbf{Domain knowledge base} (\texttt{data/college\_data.json}): structured college information (fees, exams, placements, hostel, admissions, etc.) used for retrieval of relevant facts for response grounding.
\end{itemize}

\textbf{Preprocessing.} User input and intent patterns are normalized using a dedicated preprocessing stage (\texttt{text\_preprocessor.py}). Key transformations include typo normalization and abbreviation expansion, Hinglish/code-switch mapping for common Hindi--English tokens, and regex-based cleanup for punctuation artifacts and whitespace normalization. This preprocessing targets practical robustness issues where minor orthographic noise can degrade NLU signals \cite{belinkov2018synthetic,sun2020adversarial}.

\subsection{Model Implementation: Training and Fine-Tuning}
The implementation uses pre-trained transformer models without project-specific fine-tuning of large encoders/decoders. Instead, fast-to-train components are fit on local intent patterns, while response generation relies on instruction-following LLM inference.

\subsubsection{Intent Components for Startup Training}
\begin{itemize}
	\item \textbf{Semantic classifier:} a Sentence-Transformers encoder (default: \texttt{all-MiniLM-L6-v2}) embeds each preprocessed intent pattern; inference selects the intent with maximal cosine similarity to the user embedding.
	\item \textbf{TF--IDF classifier:} a scikit-learn TF--IDF vectorizer (unigrams/bigrams) and logistic regression classifier are trained on the same preprocessed patterns.
	\item \textbf{Ensembling and calibration:} the final intent prediction uses weighted aggregation with explicit calibration heuristics (agreement boosts, scaling factors, confidence floors) to improve stability under paraphrases and short/noisy inputs.
\end{itemize}
This hybrid intent implementation reflects the practical combination of transformer representations \cite{vaswani2017attention,devlin2019bert,liu2019roberta} and efficient similarity-based intent matching \cite{casanueva2020banking77}.

\subsubsection{Emotion Signal from Pre-Trained Inference}
A cached transformer pipeline performs sentiment analysis and maps sentiment to coarse interaction emotions (e.g., neutral, stressed). Keyword-based filters reduce false positives on short information-seeking questions.

\subsubsection{LLM Response Generation by Inference Without Fine-Tuning}
Response generation is performed via an autoregressive LLM endpoint (Groq, with Gemini fallback), consistent with decoder-only pre-training and instruction-following alignment approaches \cite{radford2019gpt2,ouyang2022rlhf,openai2022chatgpt}. The implementation does not train a dialogue model from scratch; it conditions inference using structured prompts and retrieved domain facts.

\subsection{System Integration: Frontend and Backend}
\textbf{Frontend.} The Streamlit application orchestrates the end-to-end turn loop (message capture, inference calls, UI rendering) and maintains session identifiers for logging.

\textbf{Backend orchestration.} The orchestration layer instantiates and caches major components (intent, scope, emotion, prompt builder, LLM handler, and context manager) to amortize model-loading cost.

\textbf{Persistence and analytics.} A SQLite-backed logging layer records timestamped interactions with intent/confidence, emotion, scope decisions, response time, and LLM source. A small connection pool reduces per-turn database overhead.

\subsection{Key Functional Modules}
\subsubsection{Intent Classification}
Intent classification is implemented as an ensemble: semantic similarity over transformer sentence embeddings; TF--IDF + logistic regression for lexical pattern capture; and a context-aware intent refiner that adjusts ambiguous predictions using recent-turn intent continuity.

\textbf{Pipeline (pseudo-code style).}
\begin{enumerate}
	\item $u_t \leftarrow$ user\_message
	\item $u'_t \leftarrow \mathrm{preprocess}(u_t)$
	\item $(y_{\mathrm{sem}},c_{\mathrm{sem}}) \leftarrow \mathrm{semantic\_predict}(u'_t)$
	\item $(y_{\mathrm{lex}},c_{\mathrm{lex}}) \leftarrow \mathrm{tfidf\_predict}(u'_t)$
	\item $(y,c) \leftarrow \mathrm{weighted\_ensemble}(y_{\mathrm{sem}},c_{\mathrm{sem}},y_{\mathrm{lex}},c_{\mathrm{lex}})$
	\item $(y,c) \leftarrow \mathrm{refine\_with\_history}(y,c,\mathrm{history},\mathrm{emotion})$
\end{enumerate}
To support deployment realism, the intent decision is followed by explicit confidence thresholding and OOS handling aligned with benchmarks that emphasize rejection performance \cite{larson2019clinc150}.

\subsubsection{Entity and Slot Extraction}
The implementation includes lightweight entity extraction in the context manager to support multi-turn continuity rather than full named-entity recognition. Extracted entities include user name patterns and intent-specific fields (e.g., department/program hints, exam type, fee-related subtopics). Pronoun resolution uses the most recent tracked entities to reduce context loss in follow-up turns.

\subsubsection{Response Generation}
Response generation is implemented as a controlled LLM call:
\begin{itemize}
	\item \textbf{Prompt construction:} a prompt-engineering module injects detected intent, confidence, scope status, emotion, conversation history, and retrieved knowledge snippets.
	\item \textbf{Knowledge grounding:} relevant sections of the college JSON are selected as context, with fallback to generic templates when a requested field is absent.
	\item \textbf{Operational safeguards:} caching for repeated queries, throttling between calls, concurrency limiting, exponential backoff with jitter, and API-key rotation; if the primary provider fails, the system falls back to Gemini.
\end{itemize}
These controls are implemented to reduce hallucination risk and stabilize user experience under rate limits and variable latency \cite{khatri2018alexa,roller2021recipes}.

	words = text.split()

		clean_word = word.rstrip("?!")

		final_confidence = min(1.0, tfidf_weighted * self.confidence_scaling)
else:
	if sem_weighted >= tfidf_weighted:
		final_intent = sem_intent
		final_confidence = min(1.0, sem_weighted * self.confidence_scaling)
	else:
		final_intent = tfidf_intent
		final_confidence = min(1.0, tfidf_weighted * self.confidence_scaling)

final_confidence = max(self.confidence_min, final_confidence)
return final_intent or "unknown", final_confidence, detailed_scores
\end{lstlisting}

\subsection{Challenges Faced and Solutions}
\textbf{Noisy and code-mixed inputs.} Short, misspelled, or Hinglish queries can degrade lexical and embedding signals. A dedicated normalization layer and dual-model ensemble mitigate brittleness \cite{belinkov2018synthetic,sun2020adversarial}.

\textbf{Out-of-scope (OOS) behavior.} Closed-set intent classifiers are prone to overconfident misrouting for unsupported requests. The implementation includes explicit scope detection and confidence-based gating, consistent with OOS-oriented intent benchmarks \cite{larson2019clinc150,casanueva2020banking77}.

\textbf{Latency and rate limiting for LLM calls.} External LLM endpoints introduce variable response times and occasional rate limits. The implementation uses caching, throttling, concurrency limits, and retry/backoff policies, with provider fallback to maintain responsiveness.

\textbf{Multi-turn coherence.} Follow-up questions often omit explicit context (e.g., ``What about that?''). A bounded conversation history and lightweight entity tracking reduce context loss without requiring full dialogue-state tracking models.

This implementation provides the concrete basis for the empirical results reported in the following section.

\chapter{Results}
This section reports quantitative outcomes from (i) a component-level intent-classification evaluation using the repository intent patterns and (ii) an end-to-end automated test run recorded in a saved execution artifact. Metrics and interpretations follow established guidance that chatbot evaluation should separate component accuracy from overall conversational quality and should account for dataset artifacts and measurement bias \cite{maroengsit2019evaluation,gururangan2018annotation,geva2019shortcut,swayamdipta2020dataset}.

\subsection{Evaluation Metrics: Accuracy, Precision, Recall, and F1-Score}
Two metric groups are reported:
\begin{itemize}
	\item \textbf{Intent classification (component-level):} Accuracy and macro-averaged Precision/Recall/F1 across the closed intent set.
	\item \textbf{End-to-end system behavior:} pass rate on scenario tests (functional acceptance), out-of-scope (OOS) detection count, and response latency summary (min/median/avg/max).
\end{itemize}
Macro-averaged scores are emphasized because the intent set is multi-class and exhibits class imbalance; macro scores reduce the risk of majority-class dominance masking poor performance on smaller intents.

\subsection{Experimental Setup}
\textbf{Intent classification evaluation.} The evaluation uses the repository intent patterns in \texttt{data/intents.json}, which contains approximately 650 preprocessed patterns across 18 intent labels (about 36 patterns per intent on average). A stratified 80/20 split is applied with a fixed random seed (42). The same preprocessing logic used by the implementation (\texttt{text\_preprocessor.py}) is applied prior to training and evaluation.

Baselines are derived from components available in the repository:
\begin{itemize}
	\item \textbf{Majority baseline:} always predicts the most frequent training intent.
	\item \textbf{Keyword rules:} assigns the intent with the highest keyword match count (using the \texttt{keywords} fields in the intent JSON).
	\item \textbf{TF--IDF (1-NN cosine):} nearest-neighbor matching in TF--IDF space.
	\item \textbf{TF--IDF + Logistic Regression:} multi-class linear classifier over TF--IDF uni/bi-grams.
	\item \textbf{Sentence-Transformers semantic similarity:} \texttt{all-MiniLM-L6-v2} cosine similarity against training patterns, matching the repository's semantic intent logic.
	\item \textbf{Weighted ensemble:} semantic (0.75) + TF--IDF (0.25) intent selection using the decision rules implemented in \texttt{intent\_model.py}.
\end{itemize}

\textbf{End-to-end system tests.} End-to-end results are taken from \texttt{TEST\_RESULTS\_20260407\_085552.txt}, which executes 120 scenario tests spanning in-domain requests, hybrid queries, explicit OOS queries, typos, edge cases, and noisy inputs.

\subsection{Latency Breakdown: NLU vs LLM}
The end-to-end artifact reports multi-second response latencies but does not separate local NLU computation from LLM response time. The application implementation exposes an LLM timing signal (\texttt{response\_time}), enabling a decomposition of per-turn latency into local NLU, LLM inference, and fixed overhead. A local microbenchmark (CPU, after warm-up) indicates that the control layer is comparatively fast: intent-only inference has a median of 27.97 ms (mean 28.55 ms), and intent+emotion+scope has a median of 60.62 ms (mean 63.67 ms; $p95$ 83.86 ms). Consequently, the multi-second tail observed in the end-to-end artifact is primarily attributable to external LLM inference and network variability rather than local intent routing. For a student-facing assistant, median latency around 4.7 seconds is usable but not ideal for conversational flow. Practical mitigation includes token streaming (lower time-to-first-token), cache-first handling for high-frequency intents, and asynchronous UI status updates before full response generation completes.

\subsection{Statistical Strength of Scenario Tests}
The recorded end-to-end scenario result (94/120 passed) corresponds to a raw pass rate of 78.3\%. The 95\% Wilson confidence interval for this raw operational pass rate is [70.1\%, 84.8\%]. Because 14/26 failures were API/provider failures, a corrected functional estimate (94/106) is 88.7\% with a 95\% Wilson interval of [81.2\%, 93.4\%]. Reporting both values separates infrastructure reliability from core assistant logic.

This expanded 120-scenario evaluation includes systematic coverage of paraphrases, typos, borderline OOS queries, hybrid queries, emotional language, vague inputs, and true out-of-scope boundary cases, reducing measurement artifacts and providing more robust evidence of system behavior in realistic deployment conditions \cite{gururangan2018annotation,geva2019shortcut,swayamdipta2020dataset,larson2019clinc150}.

\subsection{Comparison with Baseline Methods}
The component-level comparison suggests that rule-based routing can be competitive in precision but may be limited in recall, reflecting reduced coverage for paraphrases and short queries. TF--IDF baselines improve macro recall and macro F1, supporting the use of statistical models for intent routing.

The repository implementation additionally includes a semantic similarity component (Sentence-Transformers) and a weighted ensemble. Because ensemble and semantic-only scores are identical in this split, the ensemble should be interpreted primarily as an operational reliability design choice (confidence calibration and safer routing under ambiguity) rather than a direct accuracy gain mechanism.

\chapter{Conclusion}
This work addresses the problem of delivering a reliable college-domain assistant that can route user queries to appropriate intents, handle out-of-scope (OOS) inputs conservatively, and produce helpful responses while reducing brittle behavior in real-world interaction.

\textbf{Approach.} The system is implemented as a controlled conversational pipeline that combines preprocessing, ensemble intent inference, scope enforcement, emotion and tone cues, and an LLM response layer with knowledge grounding and operational safeguards (e.g., fallback and recovery). The design reflects evidence that robust deployed assistants require both strong representations and explicit control mechanisms, particularly for OOS handling and noise resilience \cite{larson2019clinc150,casanueva2020banking77,belinkov2018synthetic}.

\textbf{Key findings.} Quantitative results on repository artifacts indicate: (i) lexical baselines achieve up to 0.620 accuracy and up to 0.562 macro F1 on the held-out split of intent patterns, while the semantic component achieves 0.783 accuracy and 0.756 macro F1 on the same split; and (ii) end-to-end scenario tests on an expanded 120-scenario benchmark achieve a raw pass rate of 78.3\% (94/120; 95\% CI [70.1\%, 84.8\%]), with 14 failures attributable to API/provider errors and a corrected functional pass estimate of 88.7\% (94/106; 95\% CI [81.2\%, 93.4\%]).
In condensed form, the main novelty is methodological: reporting raw operational reliability alongside infrastructure-corrected functional reliability, so evaluation reflects both deployment reality and core assistant logic.

\textbf{Contributions.}
\begin{itemize}
\item A repository-grounded multi-signal architecture combining local NLU control and external LLM generation, validated on 120 end-to-end scenarios.
\item A deployment-realistic evaluation protocol that reports both raw operational pass rate (78.3\%, 94/120) and corrected functional pass rate (88.7\%, 94/106).
\item A quantified reliability analysis with explicit failure attribution (14 API failures vs 12 non-API failures), OOS metrics (precision 0.444, recall 0.727), and confidence intervals for both reporting modes.
\end{itemize}

\textbf{Limitations.}
\begin{itemize}
	\item Intent evaluation is based on locally authored patterns rather than large-scale external benchmarks, and may overestimate generalization due to dataset artifacts \cite{gururangan2018annotation,geva2019shortcut}.
\item The pattern inventory is relatively shallow per class (about 36 patterns per intent on average), with low-sample intents (e.g., \texttt{unknown}, \texttt{campus\_life}, \texttt{greetings}, \texttt{gratitude}, \texttt{general\_info}) that increase class imbalance effects.
	\item The intent evaluation reflects pattern-set generalization and does not fully capture real-user distribution shift, multi-turn ambiguity, or adversarial/noisy inputs.
	\item Boundary input handling (e.g., punctuation-only and whitespace-only) remains a failure point and motivates stronger normalization and routing rules.
\end{itemize}

\textbf{Future work.}
\begin{itemize}
\item Multilingual support: extend token normalization maps in \texttt{text\_preprocessor.py} for Konkani and Marathi variants observed in the local student population, then re-evaluate OOS false positives on code-mixed queries.
\item Better context handling: add anaphora-aware follow-up resolution in \texttt{context\_manager.py} for pronoun-heavy turns and re-measure failures in the CONTEXT bucket.
\item Real-time deployment: implement streaming-first responses in \texttt{app.py} and cache-first replies in \texttt{llm\_handler.py}, then benchmark time-to-first-token and median turn latency.
\item Improved robustness: tune $\tau$/$\gamma$ against a held-out OOS calibration set and re-run the 120-scenario suite to improve OOS precision without collapsing recall.
\end{itemize}
